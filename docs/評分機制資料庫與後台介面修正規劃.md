# 台北之心闖關網頁：評分機制資料庫與後台介面修正規劃

更新日期：2026-07-11

本文件規劃「戶外雙語闖關」的評分機制，目標是讓老師能用清楚、快速、可匯出的方式掌握各組闖關表現，作為活動結束後獎勵與回饋依據。此規劃接續目前已完成的 Supabase 設定與 `001` 至 `012` migration，不直接取代既有作答、照片、錄音、審核、刪除流程。

## 1. 設計目標

1. 老師審核通過後才計分，避免學生只上傳但內容不正確就得分。
2. 每一個可評分項目都能獨立記錄，包含文字欄位、英文與中文單字配對、照片、錄音與外國朋友互動紀錄。
3. 老師後台能快速看到總分、各關卡分數、待審核數量與各組進度。
4. 老師可以整筆通過，也可以只通過或退回單一欄位、單一照片。
5. 學生端可以知道哪些任務已通過、已得分、待審核或需修改。
6. 匯出 CSV 時可以直接作為活動獎勵、排名與檢討資料。

## 2. 評分原則

### 2.1 基本規則

| 類型 | 得分條件 | 分數 |
|---|---|---:|
| 一般文字空格 | 學生有填寫，老師確認內容可接受 | 100 |
| 英文與中文單字任務 | 英文與中文成對正確，老師審核通過 | 100 |
| 照片任務 | 照片符合任務要求，老師審核通過 | 100 |
| 錄音任務 | 錄音符合任務要求，老師審核通過 | 100 |
| 與世界交朋友 | 一組外國朋友紀錄包含國家與合照，老師審核通過 | 300 |

### 2.2 不計分情況

| 狀態 | 說明 |
|---|---|
| 草稿 | 學生尚未送出或尚未同步完成，不計分 |
| 待審核 | 學生已送出，但老師尚未確認，不計分 |
| 退回修改 | 老師判定內容不符合任務，不計分 |
| 已刪除 | 老師刪除個別答案、檔案或整組資料後，不計分 |

### 2.3 關於「完成空格」的判斷

「完成空格」不等於自動得分。系統可以先偵測學生有填寫內容，產生待審核項目；老師確認內容正確或合理後，才正式給分。

例如：

| 學生填寫 | 系統狀態 | 老師審核後 |
|---|---|---|
| 英文單字有寫，但中文空白 | 不產生該組單字得分項 | 0 |
| 英文與中文都有寫，但配對錯誤 | 待審核 | 老師退回，0 分 |
| 英文與中文都有寫且正確 | 待審核 | 老師通過，100 分 |

## 3. 各關卡建議計分方式

### 3.1 228 和平公園景點任務

目前每個景點任務包含照片、核心英文單字、英文句子。

| 得分項目 | 分數 |
|---|---:|
| 任務照片通過 | 100 |
| 核心英文單字通過 | 100 |
| 英文句子通過 | 100 |

每個景點任務最高 `300` 分。若有 3 個景點任務，此區最高 `900` 分。

### 3.2 台博館本館觀察任務

一般觀察型任務比照「照片＋單字＋句子」處理。

| 得分項目 | 分數 |
|---|---:|
| 觀察照片通過 | 100 |
| 核心英文單字通過 | 100 |
| 英文句子通過 | 100 |

### 3.3 博物館英文分類卡

目前規劃為 4 個類別，每一類至少 5 個填答空格，每格包含英文與中文。

| 類別 | 每類欄位數 | 每格得分 | 每類最高 |
|---|---:|---:|---:|
| Regulatory | 5 | 100 | 500 |
| Informational | 5 | 100 | 500 |
| Safety | 5 | 100 | 500 |
| Exhibition | 5 | 100 | 500 |

此任務最高 `2000` 分。每一格必須英文與中文都正確才給 `100` 分。

### 3.4 古生物英文介紹

目前包含 5 個文字欄位與 1 個錄音檔。

| 得分項目 | 分數 |
|---|---:|
| name | 100 |
| fossil or prehistoric animal | 100 |
| when it lived | 100 |
| what it ate | 100 |
| interesting fact | 100 |
| 錄音檔通過 | 100 |

此任務最高 `600` 分。若老師希望錄音只作為佐證、不列入分數，可以在後續實作時將錄音分數設為 `0` 或不產生得分項。

### 3.5 台北車站雙語指標探索

目前設計為 5 種不同功能的雙語指標，學生可上傳 1 至 5 張照片，不要求一定拍滿 5 張。

| 指標功能 | 文字得分 | 照片得分 |
|---|---:|---:|
| 位置說明 | 100 | 100 |
| 方向引導 | 100 | 100 |
| 機器操作 | 100 | 100 |
| 到站／離站資訊 | 100 | 100 |
| 服務／安全 | 100 | 100 |

每一組指標最高 `200` 分，整個台北車站任務最高 `1000` 分。若學生只完成 3 組，最高可得 `600` 分。

「功能分類」下拉選單建議先作為輔助資料，不另外計分；若未來想增加辨識挑戰，可再將分類正確列為得分項。

### 3.6 與世界交朋友

每遇到一組外國朋友，就可以新增一筆紀錄。每筆紀錄包含外國朋友國家與合照照片。

| 得分項目 | 得分條件 | 分數 |
|---|---|---:|
| 外國朋友互動紀錄 | 國家有填寫，合照照片通過 | 300 |

若遇到 5 組外國朋友且都通過，最高可得 `1500` 分。此任務不需要英文反思，也不需要已完成訪談勾選欄位。

## 4. 資料庫修正規劃

### 4.1 新增資料表：`score_items`

建議新增一張專門記錄得分項目的資料表，而不是直接把分數塞進 `submissions` 或 `media_files`。原因是：一筆作答可能包含多個空格，一個任務也可能有多張照片，若直接放在原表會很難做細項審核與統整。

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | uuid | 主鍵 |
| `activity_id` | uuid | 活動 ID |
| `team_id` | uuid | 小組 ID |
| `mission_id` | uuid | 關卡任務 ID，可為空 |
| `source_table` | text | 來源表，例如 `submissions`、`media_files`、`world_friend_records` |
| `source_id` | uuid | 來源資料 ID |
| `item_key` | text | 得分項目代碼，例如 `keyword`、`sentence`、`station_sign_1_photo` |
| `item_label_zh` | text | 老師後台顯示用中文名稱 |
| `item_label_en` | text | 老師後台顯示用英文名稱 |
| `score_type` | text | `text_field`、`word_pair`、`photo`、`audio`、`world_friend` |
| `max_score` | integer | 最高分，通常為 `100` 或 `300` |
| `awarded_score` | integer | 實得分，未通過時為 `0` |
| `review_status` | text | `pending`、`approved`、`rejected` |
| `review_note` | text | 老師備註，可選填 |
| `reviewed_by` | uuid | 審核老師的 auth user ID |
| `reviewed_at` | timestamptz | 審核時間 |
| `created_at` | timestamptz | 建立時間 |
| `updated_at` | timestamptz | 更新時間 |

建議唯一條件：

```text
team_id + mission_id + source_table + source_id + item_key
```

這可以避免同一個欄位或同一張照片重複產生分數。

### 4.2 新增資料庫函式

| 函式 | 用途 |
|---|---|
| `sync_score_items_for_submission(p_submission_id)` | 依照作答內容產生或更新文字得分項 |
| `sync_score_items_for_media_file(p_media_file_id)` | 依照照片或錄音產生或更新媒體得分項 |
| `sync_score_items_for_team(p_team_id)` | 重新整理某一組全部得分項 |
| `approve_score_item(p_score_item_id)` | 老師審核通過單一得分項 |
| `reject_score_item(p_score_item_id, p_note)` | 老師退回單一得分項 |
| `reset_score_item(p_score_item_id)` | 老師取消審核，回到待審核 |
| `approve_submission_score_items(p_submission_id)` | 整筆文字作答全部通過 |
| `approve_media_score_item(p_media_file_id)` | 單一照片或錄音通過 |
| `get_teacher_scoreboard()` | 回傳教師後台統整表需要的資料 |

### 4.3 與既有審核函式的關係

目前已有：

| 既有函式 | 目前用途 | 評分機制後建議 |
|---|---|---|
| `approve_submission` | 將整筆作答設為通過 | 同時可批次通過該作答底下的得分項 |
| `approve_media_file` | 將照片或錄音設為通過 | 同時通過該媒體檔案的得分項 |
| `delete_submission` | 刪除單筆文字答案 | 同步刪除或歸零相關得分項 |
| `delete_media_file` | 刪除單一照片或錄音 | 同步刪除或歸零相關得分項 |
| `delete_team_data` | 刪除整組資料 | 透過 cascade 刪除該組所有得分項 |

### 4.4 RLS 規劃

| 角色 | 權限 |
|---|---|
| 學生小組 | 只能查看自己小組的得分項與審核狀態 |
| 一般教師 | 可查看本活動全部小組得分，並可審核、退回、取消通過 |
| 管理者 | 可查看與管理全部資料 |

建議學生端只能 `select` 自己小組的得分結果，不允許自行 `insert`、`update`、`delete` 分數。所有分數異動都由資料庫函式處理，避免學生端自行改分。

### 4.5 Storage policies

本次評分機制不需要新增 Storage bucket，也不需要改變檔案上傳路徑。照片與錄音的評分只依據 `media_files` 與 `score_items` 的審核狀態。

## 5. 教師後台介面修正規劃

### 5.1 新增「得分統整」區塊

教師後台上方建議新增整體概況：

| 指標 | 說明 |
|---|---|
| 參與小組數 | 目前有多少小組 |
| 全部已審核項目 | 老師已通過或退回的得分項 |
| 待審核項目 | 學生已上傳或填寫，但老師尚未審核 |
| 最高分小組 | 目前總分最高的小組 |
| 平均分數 | 所有小組平均分 |

### 5.2 新增「各組各任務得分統整表」

建議主表欄位：

| 小組 | 總分 | 待審核 | 退回 | 228 公園 | 台博館 | 古生物館 | 台北車站 | 與世界交朋友 | 狀態 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---|
| 第一組 | 2300 | 4 | 1 | 600 | 700 | 300 | 400 | 300 | 進行中 |

功能：

1. 可依總分排序。
2. 可依待審核數量排序。
3. 可篩選「全部」、「待審核」、「已通過」、「需修改」。
4. 點開小組可查看每一關細項。
5. 可下載 CSV。

### 5.3 小組展開後的細項畫面

每一組展開後，建議顯示：

```text
第一組
總分：2300
待審核：4
退回修改：1

228 和平公園
- 景點任務 1 照片：已通過，100 分
- 景點任務 1 核心英文單字：已通過，100 分
- 景點任務 1 英文句子：待審核，0 分

台北車站
- 位置說明照片：已通過，100 分
- 位置說明英文與中文：已通過，100 分
- 方向引導照片：待審核，0 分
```

### 5.4 老師審核按鈕

每個得分項目旁邊建議有：

| 按鈕 | 功能 |
|---|---|
| 通過 | 將該項目設為 `approved`，給滿分 |
| 退回 | 將該項目設為 `rejected`，給 `0` 分 |
| 取消通過 | 將該項目改回 `pending`，給 `0` 分 |
| 整筆全通過 | 將同一筆作答或同一組照片底下的得分項全部通過 |

按下「退回」時，建議可讓老師輸入簡短原因，例如「中文意思不符」、「照片不是雙語指標」、「錄音聽不清楚」。

### 5.5 CSV 匯出

建議提供兩種 CSV：

| 匯出檔 | 用途 |
|---|---|
| 得分總表 | 活動結束排名與頒獎使用 |
| 得分明細表 | 檢查每組每一項得分來源 |

得分總表欄位：

```text
小組名稱,隊伍代碼,總分,待審核數,退回數,228公園,台博館,古生物館,台北車站,與世界交朋友
```

得分明細表欄位：

```text
小組名稱,隊伍代碼,關卡,得分項目,類型,狀態,分數,審核時間,老師備註
```

## 6. 學生端介面修正規劃

學生端不需要看到全班排名，以避免活動中過度競爭；但建議顯示自己小組的任務狀態。

### 6.1 任務卡顯示

| 狀態 | 顯示文字 |
|---|---|
| 尚未填寫 | 草稿 |
| 已送出未審 | 待老師審核 |
| 通過 | 已通過，獲得 100 分 |
| 退回 | 需修改 |

### 6.2 小組總覽

學生可以看到：

```text
目前得分：1200
待審核：5 項
需修改：1 項
```

這能幫助學生知道自己還需要補哪些任務。

## 7. 實作順序建議

### 第 1 步：建立 `013` SQL

內容包含：

1. 新增 `score_items` 資料表。
2. 建立索引與 RLS。
3. 建立得分同步函式。
4. 建立審核通過、退回、取消通過函式。
5. 建立統整表查詢函式。
6. 修改既有審核與刪除函式，讓分數同步更新。

### 第 2 步：前端讀取得分資料

內容包含：

1. `teacherApi` 新增讀取 `score_items` 或 `get_teacher_scoreboard()`。
2. 教師後台資料型別新增分數資料。
3. CSV 匯出加入分數欄位。

### 第 3 步：教師後台 UI

內容包含：

1. 新增得分統整表。
2. 新增小組分數展開細項。
3. 新增通過、退回、取消通過按鈕。
4. 保留原本照片預覽、文字答案、刪除功能。

### 第 4 步：學生端狀態顯示

內容包含：

1. 任務卡顯示已得分與審核狀態。
2. 小組總覽顯示目前總分、待審核、需修改。

### 第 5 步：測試與部署

測試情境：

1. 學生填寫文字後，老師可看到待審核得分項。
2. 老師通過單一欄位後，該欄位加 `100` 分。
3. 英文與中文單字只填一邊時，不得分。
4. 台北車站只上傳 3 張照片時，只計 3 張照片與相關文字分數。
5. 與世界交朋友新增 5 組外國朋友時，每組通過後各得 `300` 分。
6. 老師刪除文字答案後，相關分數消失。
7. 老師刪除照片後，相關照片分數消失。
8. 老師刪除整組資料後，該組得分資料也消失。
9. CSV 匯出總分正確。

## 8. 建議的 `013` 實作範圍

為了避免一次修改太大，`013` 建議先完成後端基礎：

1. 建立 `score_items`。
2. 建立基本 RLS。
3. 建立 `approve_score_item`、`reject_score_item`、`reset_score_item`。
4. 建立 `get_teacher_scoreboard()`。
5. 建立媒體檔案得分同步。
6. 建立文字作答得分同步。

前端介面可以接在 `013` 成功執行後再做，避免資料庫與介面同時出錯時不容易判斷問題來源。

## 9. 需要老師確認的細節

以下項目建議在正式寫 `013` 前確認：

1. 古生物錄音是否也計 `100` 分？
2. 台北車站的「功能分類」是否只作為輔助資料，不另外計分？
3. 老師審核時是否需要「退回原因」欄位？
4. 學生端是否要顯示小組目前總分？
5. 是否要顯示全班排行榜？建議先只放教師後台，學生端不顯示。

## 10. 建議結論

建議採用「得分項目 score_items」作為核心設計。這樣可以保留目前既有的作答與媒體上傳流程，同時讓老師後台清楚呈現每組、每關、每個欄位、每張照片的得分狀況。

下一步建議先產出 `013_scoring_items.sql` 草案，確認 SQL 安全後，再執行 Supabase 設定與前端介面修改。
